# WebSocket二进制协议序列号问题分析与修复（2026-04-18）

## 📊 **问题描述**

基于2026-04-18测试结果，火山语音识别服务器响应不完整：
- ✅ **音频采集正常**：I2S音频采集正常，数据格式正确
- ✅ **WebSocket连接正常**：SSL连接成功，WebSocket握手完成
- ✅ **音频数据传输正常**：音频分块发送，服务器接收确认（log_id）
- ✅ **本地超时机制正常**：5秒响应超时检测已实现
- ❌ **服务器响应不完整**：只返回log_id，无text识别结果
- ❌ **40秒僵局**：系统等待服务器响应，40秒后超时中断
- ❌ **用户反馈**：服务器可能未收到正确序列号，导致无法返回最终结果

## 🔍 **问题链分析**

**火山客户服务排查建议（关键线索）：**
```
音频协议问题排查：
1. 音频数据是否包含正确的序列号（sequence number）
2. 序列号是否从1开始递增
3. 最后一包序列号是否为负数
4. 是否包含FLAG_LAST_CHUNK（0b0010）标志
```

**服务器响应模式分析：**
```
客户端发送音频 → 服务器接收音频数据 → 解析协议头 → 检查序列号 → 
如果序列号正确 → 返回中间确认（log_id） → 继续处理 → 返回最终结果（text）
如果序列号错误 → 返回中间确认（log_id） → 等待正确序列 → 超时无响应
```

**时间线证据：**
1. **音频发送完成**：服务器收到完整音频数据，返回log_id
2. **无text响应**：服务器等待正确序列号或结束标志
3. **本地超时触发**：5秒后无text响应，触发超时回调
4. **僵局避免**：避免40秒无响应僵局，系统快速恢复

## 🎯 **根本原因分析**

### **1. 序列号协议实现不完整（主因）**
- **原始实现**：BinaryProtocolEncoder未正确处理序列号
  - `encodeAudioOnlyRequest()`忽略sequence参数
  - `buildHeader()`未包含sequence字段
  - 缺少FLAG_SEQUENCE_PRESENT标志
  - 缺少FLAG_LAST_CHUNK标志
- **协议要求**：根据火山客服指导
  - 序列号从1开始递增
  - 最后一包序列号为负数
  - FLAG_SEQUENCE_PRESENT表示包含序列号字段
  - FLAG_LAST_CHUNK表示最后一包音频

### **2. 二进制协议头部格式错误**
- **头部大小**：无序列号时为4字节，有序列号时为8字节
- **序列号字段**：4字节大端整数
- **标志位**：
  - `FLAG_SEQUENCE_PRESENT (0b0001)`：头部包含序列号
  - `FLAG_LAST_CHUNK (0b0010)`：最后一包音频
  - `FLAG_SEQUENCE_AND_LAST_CHUNK (0b0011)`：两者都有
- **原始问题**：所有音频包使用相同头部格式，缺少序列号支持

### **3. VolcanoSpeechService序列号管理缺失**
- **音频分块发送**：将长音频分成多个100-200ms分块
- **序列号分配**：需要为每个分块分配递增序列号
- **最后一包标识**：需要识别最后一包并设置负数序列号
- **原始问题**：所有分块使用相同参数，未实现序列号递增

#### **2. BinaryProtocolEncoder.cpp头部构建修复**
- **修改位置**：`buildHeader()`函数（第128-168行）

**关键代码修改：**
```cpp
// 确定头部大小：如果有序列号，使用扩展头部
uint8_t headerSize = HEADER_SIZE_BYTES;
bool hasSequence = sequence != 0 && (flags & FLAG_SEQUENCE_PRESENT) != 0;
if (hasSequence) {
    headerSize = EXTENDED_HEADER_SIZE_BYTES;
}

// Byte 0: 版本(4 bits) + 头部大小(4 bits)
// 协议约定：头部大小字段值1表示4字节头部，2表示8字节头部（带序列号）
uint8_t headerSizeField = 1; // 默认4字节头部
if (hasSequence) {
    headerSizeField = 2; // 8字节头部（4字节基础头部 + 4字节序列号）
}
header.push_back(buildByte(PROTOCOL_VERSION, headerSizeField));

// 如果有序列号，添加序列号字段（4字节大端）
if (hasSequence) {
    writeUint32BigEndian(header, sequence);
}
```

- **无序列号头部**：4字节（版本+大小 | 消息类型+flags | 序列化+压缩 | 保留）
- **有序列号头部**：8字节（同上 + 4字节大端序列号）

- **递增规则**：1, 2, 3, ..., n-1

- **分块大小**：6400字节 = 200ms @ 16000Hz 16-bit mono
- **最后一包**：可能小于分块大小

## 🔧 **后续发现与修复（2026-04-18更新）**

基于测试日志的深入分析，发现关键协议错误：

### **1. 关键错误信息分析**
```
服务器错误: "autoAssignedSequence (2) mismatch sequence in request (6400)"
```
- **服务器期望序列号**: 2（自动分配）
- **服务器解析到的序列号**: 6400
- **巧合发现**: 6400正好是音频分块大小（CHUNK_SIZE）

### **2. 根本原因分析**
**头部大小字段计算错误**：
- **原代码**: `headerSizeField = 1`（4字节头部）或`2`（8字节头部）
- **问题**: 头部大小字段应表示实际头部字节数（4或8），而不是1或2
- **后果**: 服务器解析头部时，`headerSizeField=1`被解释为1字节头部，导致后续字段错位
- **字段错位**: payload大小字段（6400）被误解析为序列号字段

### **3. 已实施的修复**
**修改`BinaryProtocolEncoder.cpp` `buildHeader()`函数**：
```cpp
// 修复前
uint8_t headerSizeField = 1; // 默认4字节头部
if (hasSequence) {
    headerSizeField = 2; // 8字节头部
}

// 修复后  
uint8_t headerSizeField = HEADER_SIZE_BYTES; // 默认4字节头部
if (hasSequence) {
    headerSizeField = EXTENDED_HEADER_SIZE_BYTES; // 8字节头部
}
```

### **4. 协议格式验证**
**正确二进制消息结构**：
```
音频请求消息（无序列号）：
[4字节头部][4字节payload大小][音频数据]

音频请求消息（有序列号）：
[8字节头部（含4字节序列号）][4字节payload大小][音频数据]
```

**头部详细格式**：
- Byte 0: 版本(4 bits) + 头部大小(4 bits) - 头部大小应为4或8
- Byte 1: 消息类型(4 bits) + flags(4 bits)
- Byte 2: 序列化方法(4 bits) + 压缩方法(4 bits)
- Byte 3: 保留字节
- Bytes 4-7: 序列号（如果有，4字节大端）
